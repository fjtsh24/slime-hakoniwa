## グラデーション配色評価

### 実装の確認

`WorldMapPanel.tsx` の `<defs>` に以下4属性のグラデーションが定義されている：

| 属性 | 通常時（0%→100%） | ホバー時（0%→100%） |
|------|-----------------|------------------|
| fire | #ff4500 → #ff8c00 | #e03c00 → #e07800 |
| water | #1e90ff → #00bfff | #1070dd → #00a0dd |
| earth | #228b22 → #8b6914 | #1a6e1a → #6e5210 |
| wind | #87ceeb → #e0f7fa | #6ab8d8 → #c0eef5 |

`getBlendedFill()` 関数の挙動：
- 全属性ゼロ → `#e5e7eb`（gray-200）でニュートラル表示
- 支配属性あり → `url(#grad-{attr})` でグラデーション参照
- 混合タイル（支配属性なし相当）→ 従来どおり RGB 加重平均ブレンド

### 直感的な属性識別の評価

**fire（赤橙）・water（青）** の2属性はゲームの色彩慣習に沿っており、プレイヤーが初見で属性を識別できると判断する。

**wind（水色→白）** はやや淡い印象を受けるが、タイルにアイコン（💨）が併用されるため識別困難にはならない。

**earth（緑→茶）** に1点指摘する。

### [FUN-M-1] earth グラデーション終点色 #8b6914（茶）が属性の直感に反する可能性

**現状**: `#228b22`（深緑）から `#8b6914`（黄土色）へ変化。45°グラデーションなので光の当たる面が茶色く見える。

**懸念点**: 一般的なゲームで「earth/土」属性のタイルは緑（草地）または茶（土）いずれかに統一されることが多い。今回の実装はグラデーション内で「緑→茶」を混在させており、プレイヤーによっては「これは草地か泥地か？」と迷う場面が生じ得る。特に混合タイルとの区別で視認性が落ちる恐れがある。

**評価**: ゲームプレイを阻害するレベルではないが、将来的に `#8b6914` を `#5a3e0a`（濃い土色）に寄せるか、逆に `#6aaa64`（明るい草緑）に統一する方が「まったり楽しめる」体験と一致するため、Phase 8 以降での色調整を推奨する。

**優先度**: Medium（現時点でゲームプレイに支障なし）

### 季節フィルター連動

`getMapFilter()` が `hue-rotate` / `saturate` / `brightness` を組み合わせてマップ全体にかかる設計は評価できる。グラデーションはフィルターに従って季節感が反映されるため、winter時に water の青が沈んで見える等、意図しない副作用が生じる可能性がある。ただし CSS filter は視覚効果の増強であり、属性識別を完全に消すほどの変化ではないため現時点では許容範囲内とする。

## GA実装評価・追加推奨イベント

### 実装の確認

- `analytics.ts`: `initGA` / `trackPageView` / `trackEvent` の3関数を実装。`send_page_view: false` で自動ページビュー計測を無効化し、`trackPageView` を明示的に呼ぶ設計は適切。
- `main.tsx`: `VITE_GA_MEASUREMENT_ID` が設定されていて、かつ `VITE_USE_EMULATOR !== 'true'` の場合のみ `initGA()` を呼ぶ。開発環境に GA データを汚染しない構成として正しい。
- `App.tsx`: `useEffect` で `location.pathname` 変化時に `trackPageView` を呼ぶ。SPA のルーティングに対応している。

### 評価

ゲーム体験を計測する上で `trackEvent` 関数は実装されているが、現状呼び出し箇所がゼロである。ページ遷移のみ計測しても「プレイヤーがゲームに熱中しているか」は分からない。

### 追加推奨イベント（ゲーム体験の観点）

以下はゲームデザイン側として計測する優先度が高いイベントの候補である。実装は Phase 8 Analytics 設計（A8/Analytics）が担当する想定だが、この段階で要件として記録しておく。

| イベント名 | 発火タイミング | 重要度 | 理由 |
|-----------|--------------|--------|------|
| `reservation_complete` | 行動予約完了時 | 高 | 「予約」がコアループ中心。完了率が低ければUXに問題がある |
| `slime_evolution` | 進化発生時 | 高 | 長期継続の動機づけイベント。計測で進化到達率が分かる |
| `first_reservation` | 初回予約完了（新規ユーザー）| 高 | 新規プレイヤーがコアループに入ったかの最重要指標 |
| `turn_log_viewed` | ターンログパネル開封時 | 中 | ターン結果への関心度。低ければ演出の問題がある可能性 |
| `tile_clicked` | マップタイルクリック時 | 中 | マップ操作への習熟度・Move/Gather予約率の分母 |
| `setup_complete` | SetupPage でマップ作成完了時 | 高 | 初期離脱の判別。ここを離脱するとゲームが始まらない |

現在 `trackEvent` は実装済みであるため、呼び出しを追加するだけで計測可能。実装コストは低い。

## SHOULD課題のゲーム体験への影響評価

A2/Sec レビューで指摘された2件の SHOULD 課題を A1/Fun 観点で評価する。

### S-1: レートリミット未設定

`GET /worlds/:worldId/status` にレートリミットがない点。

**ゲーム体験への影響**: プレイヤー（ユーザー）の操作に直接影響しない。悪意あるリクエストによるコスト増加が将来問題になる可能性はあるが、現フェーズでは正規プレイヤーの体験は損なわれない。**影響なし**。

### S-2: slimes コレクションの全フィールド公開

認証済みユーザーが他プレイヤーのスライムデータを全フィールド取得できる点。

**ゲーム体験への影響**: 「ライブ観戦」「プレイヤープロフィール」等のソーシャル機能で他スライムを見る設計は意図的なものであり、現状の実装はゲームデザインと矛盾しない。`ownerUid` の露出はプライバシー観点の問題だが、まったり遊ぶ体験を壊すものではない。**影響なし（A2/Sec が担当する問題）**。

## 承認判定

[x] 条件付き承認（条件: FUN-M-1を Phase 8 以降の TODO として implementation_plan.md に記録すること）

### 条件の詳細

- **条件**: `earth` グラデーション（#228b22→#8b6914）の配色見直しを Phase 8 以降の TODO として記録する
- **理由**: 現時点でゲームプレイを阻害しないが、マップが拡張されタイル属性の多様性が増すにつれて視認性問題が顕在化するリスクがある。記録しておくことで将来の改善漏れを防ぐ
- **条件の実施者**: オーケストレーター（Claude Code 本体）が `implementation_plan.md` に1行追記するだけで完了

### 承認コメント

Phase 7 の主実装であるアイソメトリックSVGマップへのグラデーション導入は、タイルの「世界感」を高める正しい方向性の変更である。属性識別にアイコン（絵文字）が補助として機能しているため、配色の微細な問題がゲームプレイを壊すことはない。

GA実装はゲームに影響を与えない非侵襲的な構成であり、開発環境汚染防止の配慮も適切。`trackEvent` の呼び出し箇所追加は Phase 5〜8 の Analytics 設計フェーズで A8/Analytics が主導することを推奨する。

テストカバレッジの向上はコアループの品質担保として歓迎する。
