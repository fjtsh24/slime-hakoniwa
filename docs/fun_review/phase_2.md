# ゲームデザイン・面白さレビュー Phase 2

**レビュー担当**: A1 ゲームデザイン・面白さ管理担当
**レビュー日**: 2026-03-16
**対象フェーズ**: Phase 2（認証・ユーザー・マップ基盤）

---

## 1. 総評

Phase 2はインフラ・認証基盤フェーズであり、ゲームコンテンツ自体は意図的に最小化されている。評価の主眼は「プレイヤーがゲームに参加する入口（ユーザー登録・マップ初期化）の体験品質」と「Phase 3スライム育成への動線の明確さ」に置く。

全体として、技術的な基盤構築は着実に行われている。しかし、ゲームデザイン観点では「プレイヤーが何を期待して待てばよいか」を伝えるコンテキストが不足しており、特に SetupPage の待機体験に課題が残る。Phase 2単体での体験はほぼ「見えないところで動いている」状態であり、Phase 3実装後にこの段階を振り返ったとき、連続性の設計が問われる。

## 2. 各観点のスコアとコメント

### 2-1. ユーザー登録フロー（SetupPage）の体験品質

**スコア: 5 / 10**

#### 良い点

- マップ初期化が完了すると自動的に `/game` へ遷移する設計（`userProfile?.mapId` 監視）は、プレイヤーが余計な操作をしなくてよい点で優れている。
- ローディングスピナーが表示され、「何かが進行中」という最低限のフィードバックは存在する。

#### 問題点

- **「マップを初期化しています...」という文言は期待感を持たせない。** プレイヤーは「マップが何なのか」をこの時点では知らない。「あなただけの箱庭を作っています...」「スライムたちの住む世界を生成中...」など、ゲームコンテキストを含む文言のほうが初回体験として記憶に残る。
- **エラーメッセージが不親切かつ不安を煽る。** `!isLoading && !userProfile` のとき表示される「初期化に時間がかかっています。ページを再読み込みしてください。」は、プレイヤーの失敗感を高める。サーバー側のAuth Triggerが非同期で実行されるため、数秒のタイムラグは正常動作の範囲内であるにもかかわらず、即座にエラー扱いになるリスクがある。表示するタイミングを「10〜15秒経過後」に遅らせ、文言も「少し時間がかかっています。このまましばらくお待ちください。」程度に和らげるべき。
- **何秒待つのかが分からない。** 「通常数秒」とドキュメントには記載されているが、プレイヤーには伝わらない。プログレスバーや「通常5秒以内に完了します」の一文があるだけで体験が大きく改善される。
- **1回限りの体験なのにゲームの期待感を高める演出が皆無。** ユーザー登録は初回プレイの「最初の印象」を作る場面であり、スライムのシルエットや一言紹介など、微小な演出投資でも継続率に影響する。

### 2-2. マップ設定画面（MapSettingsPage）の体験品質

**スコア: 6 / 10**

#### 良い点

- 属性ごとの色分け（火=赤/水=青/土=黄/風=グレー）は直感的で分かりやすい。色の選択は一般的な属性イメージに沿っており、説明なしに属性を識別できる。
- ホバー時の tooltip で `(x,y) 火:0.X 水:0.X 土:0.X 風:0.X` を表示する設計は情報密度が高く、コンパクトな10×10グリッドでも情報確認が可能。
- 「ゲームに戻る」ボタンが右上に配置され、離脱導線が明確。

#### 問題点

- **10×10グリッドの視認性に限界がある。** `aspect-square` でタイルを均等配置しているが、各セルに `{tile.x},{tile.y}` の座標テキストを表示しているため、小さい画面では文字が潰れる。スマホ対応を想定すると座標テキストは tooltip のみに限定し、色のみでタイルを表現するほうが視認性が高い。
- **「タイル属性の変更は今後のアップデートで対応予定です」という記述は期待感が弱い。** 現状「見れるだけで何もできない画面」になっており、プレイヤーがこの画面を開く動機が薄い。Phase 2の段階では已むを得ないが、「この属性があなたのスライムの進化方向を決めます」といった説明文を加えることで、この制限を「知識提供の機会」に変換できる。つまり、変更できない制約をネガティブに提示するのではなく、マップ属性とスライム育成の関係性を学ぶ場面として活用する設計が望ましい。
- **Phase 2時点では、マップ設定画面に遷移するためのナビゲーションが GamePage にあるかどうかが不明確。** `/map-settings` ルートは定義されているが、GamePage からどのようにリンクされているかは確認できなかった。ナビゲーションが実装されていない場合、この画面はURLを直接入力しないとアクセスできない「隠し画面」になってしまう。

### 2-3. slimeSpecies.ts の進化後スライム evolutionConditions 修正評価

**スコア: 8 / 10**

#### 評価

Phase 1レビュー（M-2）で指摘した「進化後スライムの自己参照バグ」が `evolutionConditions: []` として修正されたことを確認した。これは技術的な正確性として正しい対応である。

#### ゲームバランス観点の評価

- `evolutionConditions: []` により、進化後の4種スライム（ファイア・アクア・アース・ウィンド）は現時点で「進化の終点」として明示される。これはゲームデザイン的に許容できる判断であり、「現時点では第2進化なし」という事実が暗黙的ではなく明示的になっている点で望ましい。
- 第2進化先の追加方針についてはデータ定義さえ追加すれば実装は容易な設計になっており、Phase 4（進化・分裂・融合）での拡張を阻害しない。
- ただし、進化後の各スライムがどんな「第2進化」を目指せるかのコンセプト設計は Phase 4 実装前に A1 がレビューする必要がある。現在のパラメータ（ファイアスライム: ATK 25・DEF 8・HPが低め）から推測すると、さらなる専門化（超高ATK・超低HP の「バーサーカー系」）か、複合属性化（火+水 = 「スチームスライム」など）が自然な方向性として考えられる。

### 2-4. Phase 2 → Phase 3 の連続性評価

**スコア: 5 / 10**

#### 良い点

- App.tsx のルーティング設計（`hasMap` による分岐）は「マップ未所持 → `/setup`」「マップ所持 → `/game`」という明確なフロー制御になっており、Phase 3でスライム育成を実装したときに `/game` の役割を拡張するだけで済む設計。
- `userProfile.mapId` という接続点が明確に定義されており、スライムとマップの関係を構築するための基盤データが整っている。

#### 問題点

- **Phase 2完了後、プレイヤーが次に何をすればよいか全く分からない。** `/game` に遷移した後の GamePage（Phase 1実装）には Phase 1 時点のUI（スライム一覧・行動予約フォーム）が表示されるが、Phase 2新規ユーザーはスライムを1体も持っていない可能性がある。スライムが0体の場合の GamePage 体験がどうなるかは Phase 3 との接続として要検討。Phase 3 実装前の状態では「空のゲーム画面」が表示されるだけになり、離脱リスクが高い。
- **マップ属性とスライム育成の接続説明がない。** マップに火属性のタイルがあることは分かるが、「それがスライムにとってどういう意味を持つのか」の説明が Phase 2 の画面内に存在しない。Phase 3 実装時に MapSettingsPage を「マップ属性がスライムの種族値成長に影響する」説明を持つ教育的な画面として整備することを推奨する。
- **ユーザー登録完了後の「ようこそ体験」がない。** 初めてマップを取得したときに「あなたの箱庭が完成しました！スライムを育て始めましょう」的なファーストランナー演出があると、Phase 3 への引き継ぎが自然になる。

## 3. 問題点まとめと改善提案

### 優先度：高

#### [H-1] SetupPage のエラー表示トリガーが早すぎる
- **問題**: `!isLoading && !userProfile` の状態はAuth Trigger実行中の数秒間でも発生しうる。この状態でエラーメッセージを表示すると、正常なフロー中にもエラーが表示される。
- **改善案**: 表示タイミングを `useEffect` + `setTimeout` で10〜15秒後に遅らせる。または、SetupPage 側でAPI呼び出し後のステート管理を行い、「Auth Triggerが発火したか確認中」という状態を明示する。

#### [H-2] Phase 3 接続時のスライム0体状態を設計する
- **問題**: Phase 2完了後に初回ログインしたユーザーが `/game` に遷移すると、スライムが0体の状態になる可能性がある。Phase 3 でスライム生成が実装されるまで、GamePage は何も表示しないか壊れた状態になりうる。
- **改善案**: Phase 3 実装時に「初期スライムを受け取る」オンボーディングフローを `/setup` → `/game` の遷移にはさむことを推奨する。SetupPage をマップ初期化待機だけでなく「初回セットアップウィザード」として設計し直すことを Phase 3 着手前に検討する。

### 優先度：中

#### [M-1] SetupPage の待機メッセージのゲームコンテキスト追加
- **問題**: 「マップを初期化しています...」は技術的な文言であり、ゲームの世界観を感じさせない。
- **改善案**: 「あなただけの箱庭を生成中...」「スライムたちの住む世界を準備しています...」といったゲームコンテキストを持つ文言に変更する。

#### [M-2] MapSettingsPage にマップ属性の意味説明を追加
- **問題**: 属性値が表示されているが、「この数値がゲームにどう影響するか」の説明がない。
- **改善案**: グリッドの上部または下部に「タイル属性はスライムが生活するエリアで種族値に影響します」といった1〜2行の説明を追加する。これにより「見るだけの画面」が「理解を深める画面」になる。

#### [M-3] GamePage への `/map-settings` リンク確認・追加
- **問題**: `/map-settings` ルートは定義されているが、GamePage からのナビゲーションが存在しない可能性がある。
- **改善案**: Phase 3 以降で GamePage を整備する際に、ヘッダー等に「マップ設定」リンクを追加する。

### 優先度：低

#### [L-1] 第2進化スライムのコンセプト事前設計
- **問題**: Phase 4 で第2進化を実装する際、進化後スライムのパラメータ設計が必要になる。現時点でのデータ定義が `evolutionConditions: []` で終わっているため、Phase 4 実装前にコンセプトを決めておく必要がある。
- **改善案**: Phase 3 完了後・Phase 4 着手前に A1 が第2進化ルートのコンセプト設計（スライム種類数・属性組み合わせ・パラメータ方向性）を別途レビューする機会を設ける。

#### [L-2] スマホ向けグリッド最適化
- **問題**: 10×10グリッドの座標テキスト `{tile.x},{tile.y}` はスマホ画面で潰れる。
- **改善案**: Phase 5（マップ描画・UI完成）のスコープで解決するが、MapSettingsPage の座標テキストはPhase 3以前に tooltip のみに変更しておくと視認性が改善する。

## 4. Phase 3 への申し送り事項（ゲームデザイン観点）

### 4-1. 初回スライム取得フローの設計
Phase 3 で「スライム生成（ワールド開始時に初期スライムを付与）」を実装する際、SetupPage もしくは GamePage に「初回スライムを受け取る」演出を組み込むことを強く推奨する。「箱庭が完成した → スライムが1体やってきた」という自然な流れにすることで、Phase 2 と Phase 3 の境目がプレイヤーに見えない形で接続される。

### 4-2. マップ属性とスライム育成の説明タイミング
Phase 3 で食事アクション・種族値変動が実装される際、MapSettingsPage の説明文も合わせて更新する。「このタイルの火属性が高いほど、火属性スライムの種族値成長が速くなります」といった具体的な説明が追加できるため、Phase 3 実装後に MapSettingsPage をレビューし直すことを推奨する。

### 4-3. ターン進行の「見える化」への準備
Phase 1 レビューで指摘した「次ターンに向けて計画を立てる体験」の強化は Phase 3 のコアになる。Phase 3 着手時に、以下を合わせて設計することを推奨する：
- スライムの現在 hunger 表示
- 次ターンで hunger がいくつになるかの予測値
- 「今予約しないと次ターンに空腹になる」というソフトなプッシュ通知

### 4-4. hunger ゲームループの初回体験設計（Phase 1 レビュー再掲）
Phase 1 から引き継いだ重要事項として、初回プレイの数ターンで「hunger が下がる → 食事を予約する → ターンが進んで回復する」というフィードバックループを自然に体験させる設計が必要。Phase 3 で行動予約 UI を完全実装する際に、この動線を意識したチュートリアル設計を同時に行うことを強く推奨する（Phase 5 に「チュートリアルフロー実装」が計上されているが、行動予約の基本フローは Phase 3 時点で体験可能にしておくべき）。

### 4-5. スライム種族選択の演出
Phase 3 で初期スライムを付与する際、「4種の進化後スライムをどれかから始める」か「基本スライムから始めて自分で進化方向を決める」かのコンセプト判断が必要になる。現在の設計では基本スライム（slime-001）から全進化ルートが開かれているが、「最初から個性のあるスライムを選べる」選択制にすることで初回の「自分のスライム感」が高まる可能性がある。Phase 3 着手前に A1 に確認を求めること。
