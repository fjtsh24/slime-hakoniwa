# ゲームデザイン・面白さレビュー Phase 3

**レビュー担当**: A1/Fun
**レビュー日**: 2026-03-17
**対象フェーズ**: Phase 3（スライム育成基本）

---

## 1. 総評

Phase 3 は「予約→放置→結果確認」という本ゲームのコアサイクルが初めてプレイヤーの手に届くフェーズである。技術的基盤（Phase 1・2）の上に、hunger バッジ・食料詳細パネル・ターンログ・予約一覧という育成体験の主要 UI が揃った点は大きな前進と評価できる。

一方で、「初めてプレイするプレイヤーが画面を見て何をすればよいか自分で気づける」という観点では、複数の設計上の隙間が残っている。特に、ターンログの情報量の不足・食料14種からの選択難度・行動予約のターン番号入力という3点は、初回体験の離脱リスクに直結する可能性があり、優先的に改善を検討すべきである。

全体として「動く・遊べる」状態には達しているが、「また明日もログインしたい」という継続動機を生み出す体験品質にはまだ改善余地がある。Phase 4 着手前に中優先度の改善をいくつか実施しておくと、後続フェーズの評価精度が上がる。

## 2. 各観点のスコアとコメント

### 2-1. ウェルカムカード体験

**スコア: 7 / 10**

#### 良い点

- 「はじめてのスライムを迎えよう！」というコピーは能動的な語感があり、初回体験として適切。Phase 2 H-2 の解決策（game_parameters.md 4-3 設計）がそのまま実装されており、設計意図に沿っている。
- ボタンラベル「スライムを呼び出す」は世界観に合致しており、「召喚」という行為がゲームの入口として機能している。
- 送信中の `disabled` 状態と「召喚中...」への文言変化により、ボタン連打を自然に防止している。
- 成功時は Firestore onSnapshot によってスライム一覧がリアルタイムで切り替わるため、「カードが消えてスライムが現れる」という体験の流れが自然。

#### 問題点

- **設計書（game_parameters.md 4-3）との文言差異がある。** 設計書では「まだスライムがいません。あなたの最初のスライムを迎えてみましょう。スライムは毎時間行動します。食事や移動を予約して、一緒に育てていきましょう。」という説明文を想定していたが、実装では「スライムを育てて進化させましょう。まず最初の1体を召喚してください。」と短縮されている。「スライムは毎時間行動します」という行動予約の意味説明が省略されており、初回プレイヤーにとって「予約して放置する」ゲームサイクルへの橋渡しが弱くなっている。
- **409 エラー時のメッセージが素っ気ない。** 「すでにスライムがいます」は正確だが、スライムが既にいるのに画面が表示される理由（Firestore の購読タイムラグ等）が想定外である場合、プレイヤーの混乱を招く。また 409 以外のエラー（ネットワーク障害等）は一律「エラーが発生しました」になっており、「もう一度試せるのか・サポートに連絡すべきか」が判断できない。設計書（game_parameters.md 4-3 エラー時仕様）では「スライムの呼び出しに失敗しました。もう一度お試しください。」と再試行を促す文言が指定されていたが、実装では単に「エラーが発生しました」のみである。
- **初回体験の「ワクワク感」演出が最小限。** ボタンを押した後に召喚エフェクトや一言メッセージ（「スライムが現れた！」等）がなく、スライム一覧がただ表示されるだけで、"初めて出会う瞬間"の記念感が薄い。

### 2-2. 食料選択 UI の体験

**スコア: 6 / 10**

#### 良い点

- select 要素の option に `{f.name}（{f.category}）` を表示しており、食料名とカテゴリを1行で確認できる設計は情報密度として適切。
- 選択した食料の `statDeltas`・`racialDeltas`・`skillGrantProb` を詳細パネルで即時表示する設計は「この食料を食べさせるとどうなるか」の計画立案を直接支援しており、ゲームデザイン観点で高く評価できる。「計画を立てる楽しさ」の実現として機能している。
- description 表示（斜体）は食料の世界観説明として機能しており、単なる数値羅列を避けている。

#### 問題点

- **14種の食料を select の縦リストから選ぶ体験は認知負荷が高い。** select は一度に3〜4件しか見えず、14件をスクロールで確認する必要がある。初回プレイヤーには「どれを選べばいいか分からない」という迷子感が生じやすい。game_parameters.md 5-3 の「育成目標別おすすめ食料ガイドライン」が画面内に一切表示されないため、この戦略情報が死んでいる。
- **カテゴリ別フィルタがなく、6カテゴリ（slime・plant・human・beast・spirit・fish）の区別がプレイ中に意識しにくい。** 選択肢の並び順がカテゴリ順であることは確認できるが、select のグループ化（`<optgroup>`）が実装されておらず、カテゴリの境界が視覚的に分かりにくい。
- **statDeltas の表示が `hp+5` のような機械的な形式になっている。** `Object.entries` で生成した `${k}+${v}` 形式は、HP・ATK・DEF・SPD・EXP の日本語ラベルに変換されていない。「hp+5」よりも「HP +5」のほうが初回プレイヤーにとって理解しやすい。
- **racialDeltas の `+0.1` 等の種族値変動が何を意味するか、この画面だけでは分からない。** 「種族値: fire+0.1」と表示されても、それが進化条件（fire >= 0.5 が必要）とどう繋がるかの文脈説明がない。

### 2-3. hunger システムの体験設計

**スコア: 8 / 10**

#### 良い点

- `s.stats.hunger < 20` で赤バッジ、`< 50` で黄バッジ、それ以外で緑バッジというしきい値設計は、game_parameters.md 2-1 の「hunger < 20: 弱って動けない」「hunger 20〜49: 休息モード」「hunger 50以上: 元気」という自律行動の閾値と完全に一致しており、**バッジの色が自律行動の状態変化と1対1に対応している**。これはゲームデザイン観点として優れた実装で、UI の色とゲームメカニクスが一致しており直感的に機能する。
- 「次ターン後: {Math.max(0, s.stats.hunger - 5)}」の予測値表示は Phase 2 申し送り（4-3）で強く推奨した機能であり、実装されたことを評価する。「今予約しないと次ターンに空腹になる」という意思決定の根拠を数値で提示できている。
- 「満腹度: {hunger}/100」という表記は分子/分母形式で上限を明示しており、スケールが直感的に把握できる。

#### 問題点

- **hunger が 0 になった場合に何が起きるかがUIから読み取れない。** ターンログに `autonomous: starving` イベントが記録されることは `TurnLogList` の `formatEvent` から分かるが、GamePage のスライム一覧カードには「空腹で動けない」状態であることのアラートが不足している。hunger が 0 のときは赤バッジの表示はされるが、「このままだとどうなるか」という具体的な警告（例: 「空腹状態です。食事を予約してください」のツールチップや注記）がない。
- **hunger バッジの表示位置が情報として埋もれがち。** スライム名の隣にインラインで表示されており、視覚的な強調が弱い。特に hunger が赤（緊急）の場合、もう少し目立つ演出（アイコンの追加・バッジサイズの強調等）があると警告としての機能が高まる。
- **「次ターン後」の予測値が food 予約との統合表示になっていない。** 既に eat アクションを予約している場合、次ターン後の hunger は `-5 + 30 = +25` となるが、この予測は現在の表示（単純な `-5` 計算）には反映されていない。予約がある場合は「次ターン後（食事予約あり）: {hunger + 25}」と区別して表示できると、より正確な計画支援になる。

### 2-4. ターンログ表示の体験

**スコア: 5 / 10**

#### 良い点

- 11種類の `eventType` に対して個別の色コード（eat=緑・move=青・rest=黄・battle_win=紫・battle_lose=赤・evolve=オレンジ等）を定義しており、ログの種別が色で瞬時に判断できる設計は評価できる。
- `orderBy('turnNumber', 'desc')` で最新が上に来る表示順は「直近に何が起きたか」を確認する目的に適している。

#### 問題点

- **「食事した（food-slime-001）」という形式が機械的すぎる。** `eventType === 'eat'` のログは `foodId` をそのまま表示しており、「スライムコア」等の食料名への変換がされていない。食料IDは内部管理用の文字列であり、プレイヤーには意味が分かりにくい。`ReservationList.tsx` では `foods.find(f => f.id === foodId)` で食料名を引くヘルパーを実装しているのに、`TurnLogList.tsx` の `formatEvent` では同じ処理が実装されていない。**同じデータ（foods マスタ）を参照しているコンポーネント間での表示品質の不統一**であり、修正優先度が高い。
- **game_parameters.md 6-3 で設計したターンログ表示テキスト例と実装が乖離している。** 設計書では「{スライム名}が{食料名}を食べた。EXP +{n}、hunger {before}→{after}」を想定していたが、実装では「食事した（food-slime-001）」のみで EXP 変化・hunger 変化が表示されない。これでは「予約した結果どうなったか」のフィードバックが不十分であり、育成ゲームとして核心的な「ステータス変化の確認」が欠けている。
- **直近20件という件数について。** 1時間ターンで1日24ターン、20件では約20時間分のログしか見えない。初期プレイヤーには十分だが、複数イベントが1ターンに記録される（eat + hunger_decrease など）場合はさらに短時間のログになる。現時点では許容範囲内だが、Phase 4（進化・分裂等の特別イベント）が加わった際に件数の見直しを検討すること。
- **セクションヘッダーが「ターンログ（直近20件）」のみで、選択中のスライム名が表示されない。** 将来スライムが複数体になったとき、どのスライムのログを表示しているかが分からなくなる。現在はシングルスライムの前提で問題ないが、拡張を見越して「{スライム名}のターンログ」形式への変更が望ましい。

### 2-5. 行動予約体験（ターン数選択）

**スコア: 5 / 10**

#### 良い点

- `min={currentTurn + 1}` の制約により、過去ターンへの予約を防止しており、バリデーションとして機能している。
- スライム選択・アクション種別・ターン番号・食料選択という情報の流れが上から下の順序で配置されており、フォームの記入順序として自然。

#### 問題点

- **「実行ターン番号」を number input で入力させるUIは初回プレイヤーには難しい。** 「ターン番号」という概念はゲームに慣れたプレイヤーには自然だが、初回プレイヤーは「今が何ターン目か」「次ターンが何番か」を即座に把握できない。TurnTimer（ターンタイマーコンポーネント）で現在のターン番号が別の場所に表示されているとしても、フォームから離れた位置での確認が必要になる。「次のターン（Turn N+1）」「2ターン後（Turn N+2）」といった相対的な選択 UI（ラジオボタン・ドロップダウン）のほうが直感的である。
- **複数ターン先の予約をまとめて入れる体験が1件ずつの入力に限定されている。** 「6ターン分まとめて食事を予約する」という game_parameters.md 2-3 が想定する「推奨6時間サイクル」の予約が、現在のUIでは6回フォームを入力・送信しなければならない。「N件繰り返し予約」機能はPhase 4以降のスコープでもよいが、改善余地として記録しておく。
- **move アクションの目標座標を `targetX`・`targetY` の数値入力のみで受け付ける設計は現時点では最低限。** マップグリッド（MapSettingsPage）を確認しながら座標を数値入力するという2画面往復の操作が必要になる。Phase 5（マップ描画）での改善が予定されているが、現時点でも「現在地: ({tileX}, {tileY})」を表示して移動先のヒントを与えるだけで体験が改善する。
- **rest アクション選択時に追加 UI がなく、food アクション時は詳細パネルがあるのとの対比で「何もしない感」が出る。** 「休息するとHPが微回復します」程度の1行説明があるとアクションの効果が分かりやすい。

## 3. 問題点まとめと改善提案

### 優先度：高

#### [H-1] ターンログの食料ID → 食料名変換が未実装

- **問題**: `TurnLogList.tsx` の `formatEvent('eat', ...)` が `foodId`（例: `food-slime-001`）をそのまま表示しており、プレイヤーには意味が分からない。`ReservationList.tsx` では `foods.find()` で食料名を解決しているのに、`TurnLogList` では未実装という不統一がある。
- **改善案**: `TurnLogList.tsx` の `formatEvent` 内で `foods.find(f => f.id === foodId)?.name` を参照し、「食事した（スライムコア）」形式に変更する。
- **影響範囲**: `TurnLogList.tsx` のみ、修正コスト小。

#### [H-2] ターンログにステータス変化量が表示されない

- **問題**: game_parameters.md 6-3 設計では「EXP +{n}、hunger {before}→{after}」の表示を想定していたが、実装では変化量が一切表示されない。「食事した（スライムコア）」だけでは「結果として何が変わったか」が分からず、育成ゲームのフィードバックループが機能しない。
- **改善案**: `TurnLog` の `eventData` に `statChanges: {hp?: number, atk?: number, exp?: number, hungerBefore?: number, hungerAfter?: number}` 等のフィールドを追加し（バックエンド実装も必要）、フロントエンドのログ表示に反映する。または eventData から読み取れる範囲（食料の statDeltas は foods マスタで既知）を使って表示を補完する方法もある。
- **影響範囲**: `TurnLogList.tsx` + バックエンドターン処理（eventData スキーマ拡張が理想）。修正コスト中程度。

#### [H-3] M-3（GamePage からの /map-settings ナビゲーション）が未実装

- **問題**: Phase 2 レビューから引き継いだ M-3 が Phase 3 完了時点でも未対応。MapSettingsPage にアクセスする方法がUIに存在しない。
- **改善案**: `GamePage.tsx` のヘッダーに「マップ設定」リンクボタンを追加する（`<Link to="/map-settings">` または `navigate('/map-settings')`）。修正コスト極小。

---

### 優先度：中

#### [M-1] ウェルカムカードの文言を設計書仕様に近づける

- **問題**: 「スライムは毎時間行動します。食事や移動を予約して、一緒に育てていきましょう。」という行動予約を説明する文が省略されており、初回プレイヤーへの行動誘導が弱い。
- **改善案**: ウェルカムカード内の説明文に「スライムは1時間ごとに行動します。食事・移動・休息を予約して、育てていきましょう。」程度の1〜2文を追加する。

#### [M-2] food 選択 UI のステータスラベル日本語化

- **問題**: 詳細パネルが `hp+5 / atk+3` 形式の機械的な文字列で表示されており、日本語ラベル（HP・ATK・DEF・SPD・EXP）に変換されていない。
- **改善案**: `ActionReservationForm.tsx` の `statLines` 生成部分に `{hp: 'HP', atk: '攻撃力', def: '防御力', spd: '素早さ', exp: 'EXP'}` マッピングを追加する。

#### [M-3] 行動予約フォームのターン番号入力をUI改善

- **問題**: number input への直接入力は「ターン番号」概念に慣れていない初回プレイヤーに負担をかける。
- **改善案**: 「次のターンに実行」「2ターン後に実行」「3ターン後に実行」... というプリセットボタン（最大5〜6個）を表示し、自由入力は「詳細設定」として折りたたむ。currentTurn をそれぞれ `currentTurn + n` に変換する。

#### [M-4] move アクション時に現在座標をヒント表示

- **問題**: 目標座標の数値入力時に、選択中スライムの現在地（`tileX`, `tileY`）が表示されず、どこから移動するかが分からない。
- **改善案**: `ActionReservationForm` の props に slime オブジェクトが既に含まれているため、move アクション選択時に「現在地: ({tileX}, {tileY})」の表示を追加する。

#### [M-5] hunger 警告の視覚的強化（hunger < 20 時）

- **問題**: hunger が 0 のときでも赤バッジのみで「空腹で動けない」状態のアラートが弱い。
- **改善案**: hunger < 20 のとき、バッジに加えて「空腹注意！食事を予約してください」の警告文をスライムリスト項目内に表示する。

### 優先度：低

#### [L-1] MapSettingsPage の属性説明とスライム種族値の具体的連携説明

- **問題**: 属性説明カードに「攻撃力・スキル発動率に影響」等の説明はあるが、「fire 属性タイルに移動すると racialValues.fire が増加し、0.5 以上でファイアスライムに進化できる」という具体的なゲームメカニクスとの接続説明がない。
- **改善案**: 属性説明カードの下部に「スライムを属性タイルに移動させると、対応する種族値が成長します。種族値 0.5 以上で対応する進化が解放されます。」程度の説明文を追加する（Phase 2 M-2 の完全対応）。

#### [L-2] ターンログ件数の拡張検討

- **問題**: 直近20件は現状適切だが、1ターンに複数イベントが記録される場合（eat + hunger_decrease + skill_grant 等）は実質的な履歴が短くなる。
- **改善案**: Phase 4 で進化・分裂等の特別イベントが増えた後に、件数を30〜50件に増やすことを検討する。現時点は20件で問題ない。

#### [L-3] rest アクション選択時の効果説明追加

- **問題**: eat では詳細パネルが表示されるが、rest ではフォーム上の説明が一切ない。
- **改善案**: rest 選択時に「休息すると hunger +10、HP が微回復します（最大HPの5%）」の説明パネルを表示する（eat の詳細パネルと同様の実装パターン）。

## 4. Phase 4 への申し送り事項

### 4-1. 第2進化コンセプト設計（Phase 2 L-1 引き継ぎ）

Phase 4（進化・分裂・融合）着手前に、第2進化スライム5種（ダークスライム・ライトスライム・ドラゴンスライム・マリンスライム・フォレストスライム）のゲームバランス上の役割と、第2進化後のさらなる進化ルートの有無を A1 が設計・レビューすること。現在の `slimeSpecies.ts` では5種が `evolutionConditions: []` で終端になっているが、Phase 4 での実装前に確定が必要。

### 4-2. 進化発生時の演出設計

Phase 4 で進化ロジックが実装された際、ターンログに `evolve` イベントが記録されるだけでは体験として弱い。「進化した！」という体験をゲームの重要なハイライトとして扱うため、GamePage での通知演出（バナー表示・音等）の設計を Phase 4 着手前に A4/FE と合意すること。進化はプレイヤーが数日間の育成の結果として迎える体験であり、適切な感動演出が継続動機につながる。

### 4-3. 複数スライム体験の設計

現在の GamePage は「選択中スライム1体のアクション予約」設計だが、Phase 4 以降で分裂によりスライムが複数体になった場合の UI 設計を先に検討しておく必要がある。特に以下の点を早期に合意すること:
- スライム一覧から選択 → 選択体のフォームが表示、という現在の設計はそのまま複数体対応できるか
- 全スライムへの一括予約（例: 全員に食事予約）のニーズが生まれるか

### 4-4. ターンログ eventData スキーマ拡張の要否確認

H-2 で指摘したとおり、ターンログのステータス変化量表示にはバックエンドの `eventData` スキーマ拡張が必要になる可能性がある。Phase 4 実装前に A3/BE・A5/DB と協議し、`turnLogs` コレクションのスキーマ変更計画を確定すること。

### 4-5. 行動予約 UI の UX 改善実施タイミング

M-3 で指摘したターン番号入力 UI の改善は、Phase 4 でスライムの状態が複雑になる前（Phase 3 → Phase 4 移行時）に実施しておくと、後続フェーズの体験評価がより正確になる。可能であれば Phase 4 着手前に A4/FE で対応することを推奨する。
