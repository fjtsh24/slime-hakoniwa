## Medium Priority（Phase 5 前に対応推奨）

### [SEC-M-1] インベントリ操作がインメモリ処理でトランザクション外

`addToInventory` / `removeFromInventory` はピュア関数（in-memory）であり、Firestore トランザクション内で実行されている。設計書（`docs/security_review/phase_4_design.md`）では SEC-H-2 として指摘済みのアーキテクチャ上の選択。

- **現状**: ターン処理全体が単一 Firestore トランザクション内で一括処理されており、インベントリも同バッチに含まれる設計
- **対応**: `docs/security.md` にインベントリ操作がインメモリ純粋関数として実装される設計判断（理由: シンプルさ・テスト容易性）を明記すること

### [SEC-M-2] `slimesToDelete`（融合削除）がメインバッチ後に実行（非アトミック）

融合で削除すべきスライムは、メイン WriteBatch の `commit()` 完了後に別途削除されている。バッチコミット成功後・削除実行前にクラッシュすると「ゴーストスライム」が残る。

- **リスクレベル**: 低（クラッシュ確率は低く、次ターンに同じ融合予約は実行されない）
- **Phase 5 対応**: 削除処理を同一バッチに含める、または Firestore Scheduled Cleanup を検討する

### [SEC-M-3] 分裂で生成される新スライムもバッチ後処理（非アトミック）

`newSlimesToCreate` も `slimesToDelete` と同様にメインバッチ後の別処理。バッチコミット成功後に新スライム生成が失敗した場合、分裂イベントが記録されているのにスライムが存在しない状態になる可能性がある。

- **Phase 5 対応**: 新スライム生成をメインバッチに組み込むことを検討する

### [SEC-M-4] Firebase App Check 未実装

Phase 3 からの持ち越し。API エンドポイントへの不正アクセス防止として引き続き推奨。
