# スライム箱庭ゲーム AIエージェント計画

## 設計方針

本プロジェクトは専門性の異なる複数のAIエージェントが**並列・協調**して開発を進める。
各エージェントは担当ドメインに責任を持ち、成果物のレビューには関連エージェントを相互に参加させる。

---

|----|--------------|------|--------------|

| A2 | セキュリティ担当 | Sec | 全フェーズ |

## 各エージェントの詳細

---

### A1: ゲームデザイン・面白さ管理担当（Fun）

**ミッション**：ゲームとして「面白いかどうか」の軸を常に保持し、技術実装がゲーム体験を損なわないよう管理する。

**役割と責務**

- ゲームループ（行動予約→ターン進行→結果確認）の体験品質を評価する
- スライムの育成・進化・融合の各パラメータ設計をレビューし、プレイヤーが達成感を感じられる難易度バランスを維持する
- ターン間隔・ハンガー（満腹度）減少速度・進化条件の数値を「まったり楽しめるか」の観点で定期的に見直す
- 新機能の実装前に「これはプレイヤーにとって何が楽しいか」を言語化し、実装方針に組み込む
- チュートリアルの設計と各段階での離脱リスクを評価する

**主な成果物**

- ゲームパラメータ設計書（`docs/game_parameters.md`）
- 各フェーズのゲーム体験評価レポート（`docs/fun_review/phase_N.md`）

**参加すべきレビュー**

- Phase 3（食事・育成パラメータ）の設計レビュー：必須参加
- Phase 4（進化・分裂・融合条件）の設計レビュー：必須参加
- Phase 5（マップUI・予約UI）のUXレビュー：必須参加
- 野生スライムのAI行動パラメータレビュー：必須参加

**他エージェントとの連携**

| 連携先 | 内容 |
|--------|------|
| BE (A3) | ターン処理ロジックにゲームデザイン上の制約を伝達する |
| FE (A4) | 行動予約UIが「計画を立てる楽しさ」を体現しているかフィードバック |
| DB (A5) | スライムステータス・種族値の数値レンジをレビューする |
| QA (A7) | ゲーム体験テスト観点（退屈にならないか、やりすぎ感がないか）を提供する |

### A2: セキュリティ担当（Sec）

**ミッション**：OWASP Top 10準拠を維持しつつ、ゲーム固有の不正行為（チート・不正予約・他プレイヤーデータへの不正アクセス）を防ぐ。

**役割と責務**

- Firestoreセキュリティルール（`firestore.rules`）の設計・レビュー・テスト
- Netlify Functions / Cloud FunctionsでのFirebase IDトークン検証の実装レビュー
- 入力バリデーション（zod）の実装確認
- 環境変数・シークレット管理の運用ルール策定
- 不正な行動予約（他プレイヤーのスライムへの操作・過去ターンへの予約等）の防止設計
- CI/CDパイプラインでのシークレット漏洩防止確認
- 定期的なセキュリティレビュー（各フェーズ終了時）

**担当するOWASP Top 10対策**

| 項目 | 具体的対策 |
|------|-----------|
| A01 アクセス制御の不備 | Firestoreルール：所有者チェック。Functions：IDトークン検証 |
| A02 暗号化の失敗 | HTTPS必須設定・環境変数による機密情報管理の徹底 |
| A03 インジェクション | zodによる全入力値バリデーション |
| A05 セキュリティ設定ミス | Firebase App Check（必要に応じて）・Firestoreインデックス最小化 |
| A07 認証の失敗 | IDトークンの有効期限検証・Firebaseセッション管理の適切な利用 |
| A09 ログ・監視の失敗 | turnLogs + Cloud Loggingへの不審アクセス記録 |

**主な成果物**

- セキュリティ設計書（`docs/security.md`）
- `firestore.rules` および `firestore.rules.test.ts`
- 各フェーズのセキュリティレビュー結果（`docs/security_review/phase_N.md`）

**参加すべきレビュー**

- Phase 1（Firestoreルール・予約API）：必須参加・実装担当
- Phase 2（認証フロー）：必須参加
- Phase 3以降（新機能追加時）：セキュリティ影響の確認

**他エージェントとの連携**

| 連携先 | 内容 |
|--------|------|
| BE (A3) | Cloud Functionsの入力検証・Admin SDK利用箇所のレビュー |
| FE (A4) | フロントでのトークン取り扱い・XSS対策レビュー |
| DB (A5) | Firestoreルールとデータ構造の整合性確認 |
| Infra (A6) | GitHub Secrets管理・CI/CDパイプラインの機密情報漏洩防止 |
| QA (A7) | セキュリティテストケース（不正アクセスシナリオ）の提供 |

## エージェント間の協調ルール

### レビュー体制

すべてのPhase完了時に以下のレビューを実施する：

```
実装完了
  → QA (A7)：テストカバレッジ確認
  → Sec (A2)：セキュリティレビュー
  → Fun (A1)：ゲーム体験レビュー（ゲームロジック変更がある場合）
  → 全員承認 → 次Phaseへ
```

### 成果物の依存関係

```
A5 (DBスキーマ・型定義)
  └→ A3 (ターン処理・API実装)
       └→ A4 (フロントエンド実装)
            └→ A7 (統合テスト)

A2 (セキュリティルール)
  └→ A3・A4 (ルールに従った実装)

A6 (CI/CD環境)
  └→ 全エージェントのデプロイ基盤
```

### Phase 1における担当分担

Phase 1は全エージェントの基盤となるため、並列で立ち上げる。

| Week | 担当 | タスク |
|------|------|--------|
| Week 1 | A5 | shared/types・DBMLスキーマ・Firestoreシードデータ |
| Week 1 | A6 | リポジトリ構成・Firebase Emulator・ci.yml |
| Week 1 | A2 | firestore.rules基本設計 |
| Week 2 | A3 | turnProcessor.ts TDD実装 |
| Week 2 | A7 | ターン処理テスト設計・統合テスト実装 |
| Week 2 | A2 | firestore.rulesテスト実装 |
| Week 3 | A3 | 予約CRUD API実装 |
| Week 3 | A4 | ログイン画面・TurnTimer・ActionReservationForm |
| Week 3 | A6 | deploy.yml・本番デプロイ確認 |
| Week 3 | A1 | Phase 1完了後のゲーム体験レビュー |

## エージェント起動の優先順位

新しい実装タスクに着手するとき、以下の順序でエージェントを考慮する：

1. **A2（Sec）**：新しいデータ操作・APIの設計段階で必ず参照
2. **A1（Fun）**：ゲームパラメータ・ユーザー体験に関わる変更時に必ず参照
3. **A5（DB）**：スキーマ変更・新エンティティ追加時
4. **A3（BE）**：ゲームロジックの実装
5. **A4（FE）**：UIの実装
6. **A7（QA）**：実装完了後のテスト・カバレッジ確認
7. **A6（Infra）**：環境変更・デプロイ設定変更時
